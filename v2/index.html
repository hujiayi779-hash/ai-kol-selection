Part 02

产品框架搭建

框架怎么定、边界怎么划、踩了哪些坑。

2-A  框架怎么确定？

不是先设计，而是先把任务直接丢给 AI，让它自由跑——反复跑了多次，每次涌现出的流程几乎一致，和运营实际作业逻辑高度吻合，这才固化为框架。

💡 方法论：AI 是很好的工作流探索工具。 让它先跑，再用人的经验验证固化——不是纯粹靠主观设计。
2-B  边界怎么划？

先全部用 AI 跑一遍，再看哪些环节又慢又不稳定——那就是该换成代码的地方。
「获取达人数据」就是典型：让 AI 模拟点击抓数据，一会超时一会漏抓，这根本不是 AI 该做的事。

🤖 AI 做

  • 🎨 理解、判断、归纳
  • 🔍 风格、调性等软性评估
  • ⚖️ 多维度综合决策

⚙️ 代码做

  • 📡 精密重复、稳定抓取
  • 🔧 接口调用、格式转换
  • 📋 规则明确的过滤逻辑
💡 判断标准：AI 是在「做」，还是在「假装做」？ 如果是后者，换成代码。
2-C   踩坑实录
坑 #1
📋 只给 AI「结论」，没给它「思维过程」

只告诉 AI 结论：报价 ≤5000、CC ≥X……结果 AI 变成了硬规则过滤器，有一条不达标就直接 pass。


真实案例：达人报价 6000 超预算，AI 直接淘汰。但运营实际会再看——内容方向匹配、阅读单价低，这种大概率还是会选。


转机：把判断逻辑讲给 AI——「评估断更，核心是看流量是否稳定，而不是卡间隔天数。」AI 立刻从规则过滤切换成了真正的判断。

💡 培养 AI 的「审美」，不是配置规则。 告诉它判断逻辑，它才能举一反三；只告诉结论，它只会过滤。

范式对比：规则配置 vs 认知传递

🗂 传统：穷举所有情况，写成条件判断
→
🧠 AI 时代：传递判断方式，让 AI 举一反三
📋 上限是规则的完备性，现实永远穷举不完
→
✨ 理解了逻辑，能处理没见过的新情况
💬 对产品人的新要求：能把隐性的判断逻辑显性化——说清楚「我为什么这样判断」，不只是「我的结论是什么」
坑 #2
🔄 让同一个 Agent 又执行又自检

让 AI 按 1→2→3→4 步执行，每步还要自检。结果 AI 经常「偷懒」——执行走过场，检查也走过场。


解法：专门设一个主 Agent 负责监督调度，每一步单独唤起子 Agent 执行。监督的只管监督，执行的只管执行。

❌ 之前：一个 Agent 包揽所有
整合规则 → 获取数据 → 分析内容 → 判断 → 自检
上下文越来越长，注意力分散，执行和检查都走过场
✅ 改进后：主控 + 子 Agent 各司其职
🎯 主 Agent（监督 · 调度 · 检查）
📋 子①
整合规则
📡 子②
数据获取
🎨 子③
内容理解
⚖️ 子④
匹配判断
💡 AI 的注意力是有限资源，架构设计要管理它。 职责分离之后，每个 Agent 上下文干净、只关注一件事，整体质量自然就上来了。
Part 03

1 → 100：规模化落地

从单次跑通到面向全业务稳定运行，要过的几道关。

1 和 100 之间，差了三件事
🏗
稳定规模执行
偶尔能跑 → 稳定线上运行
⚡
时效性
100个达人，1小时内出结果
🌐
跨业务通用性
一套产品，服务全业务线
🏗 稳定规模执行：AI 的工作脚手架

本地跑通一次 ≠ 稳定上线。研发需要在 AI 背后搭一套脚手架：入口触发 → 调度管理 → 结果输出。这是 AI 从「实验品」变成「生产工具」的关键一步。

⚡ 时效性：为每个达人开独立「工位」

串行分析 100 个达人——排队等、越往后上下文越混乱、质量越来越差。

❌ 串行排队
达人① → 达人② → 达人③…
慢、且越来越乱
✅ 并行独立工位
每个达人独立处理单元
上下文干净，互不干扰
🌐 跨业务通用性：通用框架 + 知识库个性化

每个业务选达人的标准不同——美食和旅行完全两套逻辑。解法：框架通用，知识库个性化。来新业务不用重新搭，只需维护知识库，运营直接往里输入经验。

🏗 通用框架层（固定）

  • ✓ 四步处理流程
  • ✓ 判断方式和评分机制
  • ✓ 输入 / 输出格式规范

📚 知识库层（按业务定制）

  • ✓ 业务特殊筛选标准
  • ✓ 历史选人经验和案例
  • ✓ 运营可随时补充维护
💡 目标状态：调优只改知识库，不动代码——调优权真正交到懂业务的产运手里。
核心挑战
🔧 友好型框架：还没做到
❌ 现状
改规则 → 改脚本代码 → 依赖研发 → 迭代慢
✅ 理想状态
改规则 → 只改知识库 → 产运独立完成
💡 AI 时代，研发最重要的产出之一是「友好型框架」——让不懂代码的产运能独立调优。研发从「功能实现者」升级为「赋能者」。

AI 时代，产运研协作范式变了什么？

不只是工具变了——每个角色的定位、协作结构都在重新定义。

4-A  定位变了
🧠
运营
AI 的业务老师
+ 质量评审员
过去：执行筛选，按规则操作
↓
迭代知识库 = 调教 AI
对结果做最终评审与验收
🏗
研发
AI 基础设施建设者
过去：写功能代码
↓
搭脚手架 · 封装 tool
构建友好型框架
🎯
产品
AI 工作流设计师
+ 效果诊断师
过去：写需求 → 研发实现
↓
设计 AI 怎么干活
定义人机边界 · 诊断效果
4-B  协作结构变了

过去：线性串行（产品写需求 → 研发实现 → 运营使用）
现在：研发是底层底座，搭好后退幕后；产运在上面持续快速迭代

💡 改一下知识库，下次跑就不一样了——迭代主动权交到了离业务最近的人手里。
